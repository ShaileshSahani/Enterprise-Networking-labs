🔧 Lab 16/50 — OSPF MTU Mismatch (Stuck in EXSTART)

Built an OSPF troubleshooting lab today focused on a routing issue that can be surprisingly difficult to identify in production environments.

🔥 OSPF Neighbor Stuck in EXSTART Due to MTU Mismatch

Lab Topology

• R1 ↔ R2
• OSPF Area 0
• Same subnet
• Same hello/dead timers

Configured:

• R1 MTU = 1500
• R2 MTU = 1400

At first glance everything looked correct.

━━━━━━━━━━━━━━━━━━━━━━

Observed Behavior

OSPF adjacency started forming normally:

DOWN → INIT → 2-WAY → EXSTART

Then stopped.

The routers never reached FULL adjacency.

No OSPF routes were exchanged.

━━━━━━━━━━━━━━━━━━━━━━

Troubleshooting Performed

Verified neighbor status:

show ip ospf neighbor

Checked interface MTU values:

show interface g0/0

Used:

debug ip ospf adj

Observed continuous DBD packet retransmissions caused by MTU mismatch during LSDB synchronization.

━━━━━━━━━━━━━━━━━━━━━━

Root Cause

During Database Description (DBD) packet exchange, OSPF validates MTU values between neighbors.

When MTU values don't match:

• LSDB synchronization fails
• DBD packets are rejected
• Adjacency remains stuck in EXSTART/EXCHANGE
• Routes are never installed

━━━━━━━━━━━━━━━━━━━━━━

Fix Applied

✅ Matched MTU values on both routers

Also tested:

ip ospf mtu-ignore

After correction:

• Neighbor state moved to FULL
• OSPF routes appeared successfully
• LSDB synchronization completed normally

━━━━━━━━━━━━━━━━━━━━━━

Key Learning

When OSPF is stuck in EXSTART, don't immediately assume timer or authentication issues.

MTU mismatch is one of the most common real-world causes and is frequently seen during:

• GRE/IPsec deployments
• MPLS migrations
• VMware networking changes
• Jumbo-frame environments
• WAN MTU modifications
