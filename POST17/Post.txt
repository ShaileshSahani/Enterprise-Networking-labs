🔧 Lab 17/50 — OSPF Network Type Mismatch

Built another OSPF troubleshooting lab today focused on a routing issue that can be extremely deceptive in enterprise environments.

🔥 OSPF Network Type Mismatch

Lab Setup:

• R1 using default OSPF Broadcast network type
• R2 configured as Point-to-Point
• Ethernet connection between both routers
• Same Area 0

At first glance everything appeared healthy.

━━━━━━━━━━━━━━━━━━━━━━

Observed Behavior:

✅ OSPF neighbors reached FULL state

But deeper investigation revealed:

• Different OSPF network type expectations
• DR/BDR behavior mismatch
• Inconsistent LSA handling
• Potential route advertisement issues

This is the kind of problem that can waste hours because the neighbor relationship looks completely normal.

━━━━━━━━━━━━━━━━━━━━━━

Troubleshooting Performed:

Verified neighbor status:

show ip ospf neighbor

Neighbor state appeared healthy:

FULL/DR

Then checked interface details:

show ip ospf interface g0/0

Found:

• R1 → BROADCAST
• R2 → POINT-TO-POINT

━━━━━━━━━━━━━━━━━━━━━━

Root Cause:

Broadcast networks expect:

• DR/BDR elections
• Multi-access behavior

Point-to-Point networks expect:

• Direct adjacency only
• No DR/BDR election

When network types don't match, OSPF behavior becomes inconsistent even if the adjacency reaches FULL state.

━━━━━━━━━━━━━━━━━━━━━━

Fix Applied:

Tested both approaches:

• Both routers configured as Broadcast
• Both routers configured as Point-to-Point

After clearing the OSPF process:

✅ Stable adjacency
✅ Consistent route exchange
✅ Matching OSPF behavior
✅ Proper LSA synchronization

━━━━━━━━━━━━━━━━━━━━━━

Key Learning:

A FULL OSPF adjacency does not always mean the network is healthy.

Real troubleshooting requires validating:

• Neighbor states
• Interface parameters
• OSPF network types
• LSA behavior
• Route propagation

━━━━━━━━━━━━━━━━━━━━━━

Common real-world causes:

• MPLS migrations
• Metro Ethernet deployments
• Tunnel conversions
• Datacenter interconnects
• Copy-pasted configurations from different link types

Understanding why protocols behave a certain way is often more important than simply making them work.
