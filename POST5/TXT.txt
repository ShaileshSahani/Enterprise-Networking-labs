🔧 Lab 05/50 — EtherChannel Mismatch Catastrophe

Today’s lab was a good reminder that:

Layer 2 problems become ugly very fast.

━━━━━━━━━━━━━━━━━━━━━━

Built an EtherChannel between two Cisco switches using dual uplinks.

Then intentionally misconfigured it.

One side:
• PAgP desirable

Other side:
• mode on

━━━━━━━━━━━━━━━━━━━━━━

Result:

• Interfaces failed to bundle properly
• STP topology became unstable
• Syslog errors started flooding
• Links behaved unpredictably

Exactly the kind of issue that creates random packet loss in production and wastes hours during troubleshooting.

━━━━━━━━━━━━━━━━━━━━━━

Verification Performed:

• EtherChannel state analysis
• STP behavior validation
• Syslog monitoring
• Port-channel status verification
• Interface consistency checks

Observed classic errors such as:

• channel misconfiguration
• interfaces suspended
• cannot bundle ports

━━━━━━━━━━━━━━━━━━━━━━

Recovery Phase

Rebuilt the topology correctly using:

LACP active/active

Immediately stabilized the network.

Both physical interfaces bundled cleanly into a single logical Port-Channel.

STP now detected only one logical path instead of multiple competing Layer 2 links.

━━━━━━━━━━━━━━━━━━━━━━

Most tutorials explain EtherChannel as:

“Combine links for more bandwidth.”

This lab focused on:

What actually happens when negotiation protocols don’t match —
and why consistency matters more than memorizing commands.
