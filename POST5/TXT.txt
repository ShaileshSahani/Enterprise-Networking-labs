Lab 05/50 — EtherChannel Mismatch Catastrophe

Today’s lab was a good reminder that:
Layer 2 problems become ugly very fast.

━━━━━━━━━━━━━━━━━━━━━━

Built an EtherChannel between two switches using dual uplinks.

Then intentionally misconfigured it.

One side:
PAgP desirable

Other side:
mode on

━━━━━━━━━━━━━━━━━━━━━━

Result:

• Interfaces failed to bundle properly
• STP topology became unstable
• Syslog errors started flooding
• Links behaved unpredictably

Exactly the kind of issue that creates random packet loss in production and wastes hours during troubleshooting.

━━━━━━━━━━━━━━━━━━━━━━

Checked:
• EtherChannel states
• STP behavior
• Syslog messages
• Port-channel status

Saw classic errors like:
• channel misconfiguration
• interfaces suspended
• cannot bundle ports

━━━━━━━━━━━━━━━━━━━━━━

Then rebuilt the entire thing correctly using:

LACP active/active

Immediately stabilized the topology.

Both physical links bundled cleanly into a single logical Port-Channel.

STP now saw only one logical path instead of multiple competing links.

━━━━━━━━━━━━━━━━━━━━━━

Most tutorials explain EtherChannel like:
“Combine links for more bandwidth.”

This lab shows:
What happens when negotiation modes don’t match.

And why proper consistency matters more than the commands themselves.

• Debug output

• Working LACP setup

Everything documented.
