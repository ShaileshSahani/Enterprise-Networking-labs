Lab 01/50 — STP Root Bridge Manipulation & Path Analysis

Built a 6-switch redundant Layer 2 topology today using Cisco 2960 switches.

No static root bridge configuration initially.
Let STP elect everything naturally first.

━━━━━━━━━━━━━━━━━━━━━━

Topology:
• 6 switches in a ring
• Multiple redundant paths
• Dual uplinks between core/distribution
• VLAN 1 + VLAN 10 trunked everywhere

Exactly the kind of topology where loops become dangerous very quickly.

━━━━━━━━━━━━━━━━━━━━━━

Phase 1:
Natural STP election.

No priorities configured.

Observed:
• Lowest MAC address became Root Bridge
• STP blocked one redundant path automatically
• Entire topology recalculated around the elected root

Then mapped:
• Root ports
• Designated ports
• Blocked ports
• Actual forwarding path through the ring

━━━━━━━━━━━━━━━━━━━━━━

Phase 2:
Forced S1 to become Root Primary.

Configured:
• S1 = Root Primary
• S2 = Root Secondary

Waited for convergence and watched the entire Layer 2 topology shift.

━━━━━━━━━━━━━━━━━━━━━━

What changed:
• Root ports moved
• Blocked ports changed location
• Some forwarding links became blocking
• Traffic paths completely changed

That’s the part most people never actually test.

STP isn’t just:
“Loop prevention.”

It’s:
“How the entire Layer 2 network decides where traffic flows.”

━━━━━━━━━━━━━━━━━━━━━━

Verification included:
• show spanning-tree analysis
• Root bridge validation
• Blocked-port mapping
• MAC path tracing
• Continuous ping testing
• Convergence validation

No packet loss after stabilization.
Clean forwarding topology.

━━━━━━━━━━━━━━━━━━━━━━

Most networking tutorials stop at:
“STP elects a root bridge.”

This lab shows:
What happens to the entire network when the root changes.
